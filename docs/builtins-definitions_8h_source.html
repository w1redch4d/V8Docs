45// Builtins are additionally split into tiers, where the tier determines the
46// distance of the builtins table from the root register within IsolateData.
47//
48// - Tier 0 (T0) are guaranteed to be close to the root register and can thus
49// be accessed efficiently root-relative calls (so not, e.g., calls from
50// generated code when short-builtin-calls is on).
51// - T1 builtins have no distance guarantees.
52//
53// Note, this mechanism works only if the set of T0 builtins is kept as small
54// as possible. Please, resist the temptation to add your builtin here unless
55// there's a very good reason.

282 /* Trampolines called when returning from a deoptimization that expects */ \
283 /* to continue in a JavaScript builtin to finish the functionality of a */ \
284 /* an TF-inlined version of builtin that has side-effects. */ \
285 /* */ \
286 /* The trampolines work as follows: */ \
287 /* 1. Trampoline restores input register values that */ \
288 /* the builtin expects from a BuiltinContinuationFrame. */ \
289 /* 2. Trampoline tears down BuiltinContinuationFrame. */ \
290 /* 3. Trampoline jumps to the builtin's address. */ \
291 /* 4. Builtin executes as if invoked by the frame above it. */ \
292 /* 5. When the builtin returns, execution resumes normally in the */ \
293 /* calling frame, processing any return result from the JavaScript */ \
294 /* builtin as if it had called the builtin directly. */ \
295 /* */ \
296 /* There are two variants of the stub that differ in their handling of a */ \
297 /* value returned by the next frame deeper on the stack. For LAZY deopts, */ \
298 /* the return value (e.g. rax on x64) is explicitly passed as an extra */ \
299 /* stack parameter to the JavaScript builtin by the "WithResult" */ \
300 /* trampoline variant. The plain variant is used in EAGER deopt contexts */ \
301 /* and has no such special handling. */ \
