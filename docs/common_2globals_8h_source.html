263// V8_HEAP_USE_PTHREAD_JIT_WRITE_PROTECT controls how V8 sets permissions for
264// executable pages.
265// In particular,
266// 1) when memory region is reserved for code range, the whole region is
267// committed with RWX permissions and then the whole region is discarded,
268// 2) since reconfiguration of RWX page permissions is not allowed on MacOS on
269// ARM64 ("Apple M1"/Apple Silicon), there must be no attempts to change
270// them,
271// 3) the request to set RWX permissions in the execeutable page region just
272// commits the pages without changing permissions (see (1), they were already
273// allocated as RWX and then deommitted),
274// 4) in order to make executable pages inaccessible one must use
275// OS::DiscardSystemPages() instead of using OS::DecommitPages() or setting
276// permissions to kNoAccess because the latter two are not allowed by the
277// MacOS (see (2)).
278//
279// This is applicable only to MacOS on ARM64 ("Apple M1"/Apple Silicon) which
280// has a APRR/MAP_JIT machinery for fast W^X permission switching (see
281// pthread_jit_write_protect).
282//
283// This approach doesn't work and shouldn't be used for V8 configuration with
284// enabled pointer compression and disabled external code space because
285// a) the pointer compression cage has to be reserved with MAP_JIT flag which
286// is too expensive,
287// b) in case of shared pointer compression cage if the code range will be
288// deleted while the cage is still alive then attempt to configure
289// permissions of pages that were previously set to RWX will fail.
290//
291#if V8_HAS_PTHREAD_JIT_WRITE_PROTECT && \
292 !(defined(V8_COMPRESS_POINTERS) && !defined(V8_EXTERNAL_CODE_SPACE))
293#define V8_HEAP_USE_PTHREAD_JIT_WRITE_PROTECT true
294#else
295#define V8_HEAP_USE_PTHREAD_JIT_WRITE_PROTECT false
296#endif
