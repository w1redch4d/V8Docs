911// This is an RAII helper class to emit a store-store memory barrier when
912// publishing objects allocated in the shared heap.
913//
914// This helper must be used in every Factory method that allocates a shared
915// JSObject visible user JS code. This is also used in Object::ShareSlow when
916// publishing newly shared JS primitives.
917//
918// While there is no default ordering guarantee for shared JS objects
919// (e.g. without the use of Atomics methods or postMessage, data races on
920// fields are observable), the internal VM state of a JS object must be safe
921// for publishing so that other threads do not crash.
922//
923// This barrier does not provide synchronization for publishing JS shared
924// objects. It only ensures the weaker "do not crash the VM" guarantee.
925//
926// In particular, note that memory barriers are invisible to TSAN. When
927// concurrent marking is active, field accesses are performed with relaxed
928// atomics, and TSAN is unable to detect data races in shared JS objects. When
929// concurrent marking is inactive, unordered publishes of shared JS objects in
930// JS code are reported as data race warnings by TSAN.
931class V8_NODISCARD SharedObjectSafePublishGuard final {
932 public:
933 ~SharedObjectSafePublishGuard() {
934 // A release fence is used to prevent store-store reorderings of stores to
935 // VM-internal state of shared objects past any subsequent stores (i.e. the
936 // publish).
937 //
938 // On the loading side, we rely on neither the compiler nor the CPU
939 // reordering loads that are dependent on observing the address of the
940 // published shared object, like fields of the shared object.
941 std::atomic_thread_fence(std::memory_order_release);
942 }
943};
